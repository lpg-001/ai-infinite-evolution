# San Skill - 用户输入语言规则和AI思考规则

## 功能说明

San Skill专注于：
1. **用户输入语言规则**：识别和理解用户的输入模式、表达方式、语言习惯
2. **AI思考规则**：定义AI如何思考、分析、处理问题的规则

## 规则分类

### 用户输入语言规则

记录用户的：
- 语言模式（中文、英文、混合等）
- 表达习惯（简洁、详细、技术性等）
- 常用术语和缩写
- 输入格式偏好

### AI思考规则

定义AI的：
- 思考流程和步骤
- 问题分析方法
- 优先级判断标准
- 响应风格和格式

## 实际应用规则

### 用户输入语言规则（已学习）

1. **升级请求表达**
   - 用户使用"全面升级"、"全面改进"等表达时，表示需要大幅度的、系统性的改进
   - "全面升级"包括：增加详细说明、完善流程、添加新功能、优化结构等
   - 用户可能使用"@文件名"来指定要操作的文件

2. **文件操作表达**
   - 用户使用"@文件名"格式指定文件时，需要精确匹配文件路径
   - 用户可能直接说文件名，也可能使用相对路径或绝对路径

3. **前端开发请求表达**
   - 用户使用"制作"、"创建"、"设计"等动词时，表示需要从零开始构建
   - 用户使用"极为精美"、"精美"等形容词时，表示需要高质量、现代化的设计
   - 用户要求"制定详细的计划并完成"时，表示需要先规划再执行，确保完整实现
   - 用户可能同时要求计划文档和实际代码实现

### AI思考规则（已学习）

1. **全面升级的思考流程**
   - 理解"全面升级"的含义：不是简单修改，而是系统性改进
   - 分析现有内容：先读取现有文件，了解当前状态
   - 参考相关文档：查看系统总览、更新机制等文档，确保升级符合系统架构
   - 扩展内容维度：增加工作流程、错误处理、性能优化、扩展性等
   - 保持一致性：确保升级后的内容与系统其他部分保持一致

2. **文件升级策略**
   - 先读取现有文件和相关文档，全面了解上下文
   - 保持原有核心功能，在此基础上扩展
   - 增加详细说明和示例，提高可读性
   - 添加最佳实践和扩展方向
   - 保持markdown格式规范

3. **前端开发任务的思考流程**（必须使用分段思考）
   - **强制要求**：前端开发任务（特别是创建1000+行代码的页面）必须使用分段思考，不能一次性思考整个任务
   - **分段思考方法**：
     - 第1段：理解需求 + 制定计划（思考：需求分析、技术栈选择、页面结构设计）
     - 第2段：HTML基础结构（思考：HTML骨架、导航、英雄区结构）
     - 第3段：核心功能模块（思考：特性展示、服务介绍、作品集等模块）
     - 第4段：CSS样式设计（思考：渐变背景、动画效果、响应式布局）
     - 第5段：JavaScript交互（思考：滚动动画、表单验证、交互反馈）
     - 第6段：优化和收尾（思考：性能优化、响应式测试、文档编写）
   - **执行原则**：
     - 每段思考后立即执行相应操作，不必等待全部思考完成
     - 保持段与段之间的逻辑连贯性，确保整体思路一致
     - 使用Todo管理跟踪每个阶段的进度

4. **多段思考策略**（强制规则，不是可选的）
   - **核心原则**：当处理长文件或复杂任务时，必须使用多段思考，禁止一次性思考整个任务
   - **为什么必须分段**：
     - 一次性思考过长会导致思考降速，影响响应速度
     - 分段思考可以提高思考速度，保持思考质量
     - 可以边思考边执行，提高整体效率
   - **分段方法**：
     - 将长文件按逻辑结构分成多个段落（如按章节、按功能模块等）
     - 每次思考一个段落，完成后再思考下一段
     - 每段思考后可以立即执行相应操作，不必等待全部思考完成
     - 保持段与段之间的逻辑连贯性，确保整体思路一致
   - **适用场景**（必须使用分段思考）：
     - 处理长文档（超过500行或包含多个章节）
     - 处理复杂任务（涉及多个步骤或模块）
     - 前端开发任务（特别是创建1000+行代码的页面）
     - 需要快速响应的场景

5. **更新机制遵循**
   - 对话结束后必须更新san/SKILL.md
   - 明确记录用户的语言模式和AI的思考方式
   - 即使没有新内容，也要进行优化性更新
